# Unit U-003 — Create UserRepository (JpaRepository, findByUname)

## Goal

Create `UserRepository extends JpaRepository<Users, Long>` with a derived `findByUname(String)` query method — the login lookup that replaces newmojf's `users_Service.findUserAccount(uname)`.

## Context

- **Vault source:** `02-architecture.md` §Backend (UserRepository component); `04-flows.md` F-U-001 step `UserRepository.findByUname(uname) → Users`.
- **Binding refs:** C-005.
- **Why this scope:** Spring Data JPA generates the impl at startup from the derived-query method name — no JPQL/native SQL needed (pack idiom; avoids the SQL-injection bypass). Depends on U-002 (entity exists).

## Implementation steps

1. Create `src/main/java/com/coresystem/coresystembackend/repository/UserRepository.java`.
2. `public interface UserRepository extends JpaRepository<Users, Long>` (jakarta namespace inherited).
3. Add `Optional<Users> findByUname(String uname);` — derived query (Spring Data parses `findBy<Uname>` → `where uname = ?`).
4. `./mvnw compile` to confirm.

## Acceptance criteria

- `UserRepository.java` compiles; extends `JpaRepository<Users, Long>`.
- `findByUname(String)` returns `Optional<Users>` (null-safe — controller handles empty).
- No raw `EntityManager` / native SQL (pack idiom).

## Anti-patterns

- DO NOT use `EntityManager.createNativeQuery` for this lookup (pack forbidden: raw SQL for standard CRUD — use derived query).
- DO NOT return the entity from a controller — this repo is consumed by the service/controller which maps to DTO (constitution §C-002, D-006).
